## 개요

기존 발굴 시스템은 다수의 위기정보를 융합해 위험가구를 선제적으로 발굴하지만, 산출된 명단이 **왜 위험한지**를 충분히 설명하지 못해 현장 담당자의 신뢰·활용으로 이어지지 않는 한계가 있습니다. 본 데모는 다음 세 지점에 집중합니다.

1. **재해석** — 단일 임계값(OR) 판정이 아니라 여러 신호의 가중 조합으로 위험을 평가하여, 개별 신호는 기준 미달이지만 동시에 위험 신호가 누적된 가구를 후보로 포착합니다.
2. **집단 특화** — 전국 단위 범용 모형 대신 대상 집단(독거노인 / 중장년 1인가구)별로 가중치를 분기하여, 집단 특수성을 반영합니다.
3. **활용도** — 점수와 신호를 근거로 한 자연어 설명·권고·우선순위·시계열 추적을 제공하여, 명단을 ‘행동’으로 전환합니다.

개선 지점은 동등한 과제가 아니라 **③ 활용도가 본체이고 ①·②는 이를 뒷받침**합니다. 화면은 4개 탭으로 이 위계를 반영하며, 활용도의 종착점으로 **방문 동선**(오늘 누구를, 어떤 순서·경로로)을 제시합니다.

## 주요 기능 (4개 탭)

상단에 **수원 모녀 사건 도입 훅**과 **증명 경계 패널**(증명/비증명/시뮬·합성)을 상설 배치합니다.

### 탭 1 — 분석 (받침 ①·②)

중복제거 요약 → **투명 점수 카드**(슬라이더 실시간 재계산 + 집단 모형 전환 + **무가중 FI ↔ 집단 가중 토글**) → **임계값 vs 조합** 대조 → **결합 급락(ML 다변량 이상탐지)**: 개별 신호는 임계 미만이지만 동시에 꺾이는 급락을 ML이 포착하는 사례를 시각화합니다. 근거 출처는 [REFERENCES.md](./REFERENCES.md)로 연결됩니다.

### 탭 2 — 활용도 Before / After (본체 ③) ★

동일한 명단을 **점수만 있는 표**(Before)와 **근거·출처 칩·추세·우선순위 번호·권고·반대근거가 결합된 카드**(After)로 대조합니다. 같은 데이터가 정보 구조에 따라 ‘행동 가능’해짐을 보여줍니다.

### 탭 3 — 방문계획 (③ 종착점) ★

**투명 융합 우선순위**(위험점수 + **급속악화(ML)** + 추세상승 + 반복통보)로 정렬하고, **궤적 군집**(급속 다변량 악화 / 만성 고위험 / …)으로 그룹화한 뒤, **근접 권역**으로 묶어 **방문 동선**(최근접 이웃 + 2-opt)을 지도에 번호 경로선으로 제시합니다. ML은 우선순위·선별까지만, 순서·포함 여부는 담당자가 최종 결정합니다.

### 탭 4 — 케이스 관리 (운영 콘솔)

KPI 스탯, 검색·필터·CSV 내보내기, ‘오늘의 업무’ 워크리스트, 다중선택 일괄작업, 케이스 상세(기여도·근거·시계열·조치 기록·상태 워크플로우)를 제공합니다.

## 아키텍처

### 설계 개요

- **클라이언트 전용 SPA** — 서버·API 키가 필요 없어 임의의 정적 호스팅에 배포 가능합니다.
- **도메인 로직과 프레젠테이션 분리** — 점수·우선순위·근거 등 핵심 로직은 `src/lib`의 순수 함수로 구현되어 UI(`src/components`)와 독립적으로 테스트·재사용됩니다.
- **결정론적 파이프라인** — 동일 입력에 대해 항상 동일 출력을 반환합니다. 점수 모형과 근거 생성 모두 부수효과가 없습니다.

### 데이터 흐름

```
scripts/generate-data.mjs        (시드 고정 → 합성 74가구 + 신호별 8주 궤적 signalHistory)
        │
        ▼
scripts/run_ml.py                (오프라인 ML: Isolation Forest·변화점·군집 → ml 필드 보강)
        │
        ▼
src/data/households.json         (정적 데이터셋: 신호·좌표·signalHistory·ml(anomaly/cluster/…))
        │
        ▼
src/lib/*  ── scoring(FI 2단) · profiles · priority(투명 융합) · ml · routing · reason · dedup · history · auth · references
        │
        ▼
src/components/*                 (React: 4탭 + ml/* 시각화 + auth/* + BottomNav · 반응형/PWA)
        │
        ▼
       UI
```

### 근거 서술 (LLM 역할 — 점수·판정 생성 안 함)

데모에서는 LLM 호출을 **결정론적 템플릿**(`src/lib/reason.ts`)으로 시뮬레이션합니다. *이미 산출된* 점수·신호·케이스 유형으로부터 자연어 근거, 권고, 반대 근거·불확실성, 근거 출처(신호 칩)를 구조화된 형태로 *서술만* 합니다. 실서비스 적용 시 이 레이어를 LLM API로 대체하되, 다음 제약을 유지하도록 설계되어 있습니다.

- 점수는 클라이언트의 투명 모형이 계산해 입력으로 전달하고, LLM은 점수를 재계산하지 않습니다(설명·서술만).
- API 키는 서버 측에서만 사용하고 브라우저에 노출하지 않습니다.
- 호출 실패 시 시뮬레이션 결과로 폴백합니다.

## 위험 점수 모형

점수의 **형식**은 임상 노쇠지수(**Frailty Index, 결핍누적**)에서, 신호의 **선택**은 한국 복지 발굴 **위기정보 체계·실서비스**(한전·통신3사 ‘AI 안부살핌’)에서 정당화합니다. 모든 상수는 *근거/가정/학습* 중 하나로 분류·주석화했습니다. 자세한 출처·강도·한계는 [REFERENCES.md](./REFERENCES.md)를 참고하세요.

각 신호를 0~1 **결핍(deficit)** 으로 정규화합니다(Rockwood 결핍누적과 동형).

```
결핍 정규화 (분모 임계는 가정):
  power     = min(전력 사용 급감률 / 100, 1)
  medical   = min(진료 단절 일수 / 365, 1)
  mail      = min(우편 미수령 주 / 4, 1)       ← ⚠️ 보조신호(확립된 근거 없음)
  telecom   = min(통신 연체 개월 / 3, 1)
  isolation = (복지관 6개월 이용 0회) ? 1 : 0
```

**2단 구조 — 베이스라인(무가중 FI) + 집단특화 확장:**

```
(1) 무가중 FI 베이스라인 (가중치 없음 = 근거 기반 출발점)
    riskBase = round( 100 × mean(deficit_i) )        ← 고전 Frailty Index(결핍 평균)

(2) 집단특화 가중 (헤드라인 점수, 차별점 ②)
    riskWeighted = round( 100 × Σ(deficit_i × w_i) / Σ w_i )
```

UI(분석 탭)에서 **무가중 FI ↔ 집단 가중**을 토글로 비교할 수 있어, “가중치를 막 정했다”는 비판을 투명하게 해소합니다. 가중치는 임의값이 아니라 **근거 강도가 상한을 제약**합니다(강=높은 가중 허용, 약=낮은 가중 강제).

**집단별 가중치 프로파일** (합 100, 근거 강도 제약) — 동일 가구라도 적용 모형에 따라 점수가 달라집니다.

| 신호 | 근거 강도 | 독거노인 (elderly) | 중장년 1인가구 (middleaged) |
| --- | --- | --- | --- |
| 전력 급감 (power) | 강 (실서비스+논문) | 28 | 30 |
| 진료 단절 (medical) | 중 | 30 | 14 |
| 통신 연체 (telecom) | 중 | 12 | 24 |
| 사회적 고립 (isolation) | 중 (이분지표는 약) | 18 | 22 |
| **우편 미수령 (mail)** | **약 (보조신호)** | **12** | **10** |

> ⚠️ **우편(mail)** 은 위기정보·실서비스 어디에도 쓰이지 않아 **확립된 근거가 없습니다.** 최저 가중을 강제하고 UI에서 “보조신호”로 구분하며, **가짜 인용을 달지 않습니다.**
> ⚠️ **사회적 고립** 은 메타분석상 강한 개념이나 *단순 이분지표(혼자/0회)는 효과가 약함*(Holt-Lunstad: OR 1.19, CI 0.99–1.44 비유의) → 과도 가중을 피했습니다.

**위험 구간(밴드)** — 임의값 65/45를 분위수 기반으로 교체:

```
고위험 ≥ 80   (합성 잔여 모집단 상위 약 38% 수준 — compute-bands.mjs 분위수 참고한 운영 조정값)
주의   ≥ 50   (조합 발굴 임계와 정렬 — "주의 이상 = 발굴 후보")
```

> 복지부 발굴 시스템도 “위기 가능성 상위 약 4%”식 **분위수 선별**을 쓰는 것이 제도적 선례입니다(데모 모집단은 위험 케이스로 큐레이션돼 4%가 아닌 데모용 컷 사용). 실제 컷 산출은 `scripts/compute-bands.mjs`(상위25%→83·상위55%→44)를 참고했고, 80/50은 “고위험은 상위 티어, 주의는 발굴 임계”라는 운영 기준으로 조정했습니다.

**발굴 방식 비교**

- 임계값 방식(기존): 단일 신호가 임계 초과 시 발굴 (`power≥80 || medical≥365 || mail≥4 || telecom≥3`)
- 조합 방식(본 데모): 가중합 점수 ≥ 50 이면 발굴 — 개별 신호는 임계 미만이지만 누적된 가구를 회수

## ML 보조선별 (작업 B)

> 심사단 지적 ②: *“AI 에이전트를 왜 쓰나? 공무원이 점수·그래프 보면 되잖아.”* → **‘에이전트’ 표현·구조를 전부 제거**하고, ML을 *판정자*가 아니라 **대량 선별기(triage)** 로 재정의했습니다.

사람이 전수로 못 보는 대량 가구(독거노인 수십만~수백만)에서 ML이 두 가지를 합니다 — **(1) 급속·다변량 악화 자동 선별**, **(2) 궤적 그룹화(우선순위 그룹)**. ML은 위험을 **판정하지 않습니다**(그건 투명 점수 + 사람). **건초더미를 줄여** 적은 인력이 집중할 곳을 가리킬 뿐입니다.

> **실서비스 선례:** 한국전력·통신3사 ‘AI 안부살핌’이 *바로 이 일*을 합니다 — 전력·통신 패턴을 개인별로 학습해 변화 신호를 자동 포착·알림. 본 ML 방향은 이미 공공에서 운영 중인 접근의 교육용 PoC입니다.

**파이프라인(`scripts/run_ml.py`, scikit-learn) — 오프라인 학습 → JSON 산출:**

| 기능 | 모델 (출처는 [REFERENCES.md](./REFERENCES.md)) |
| --- | --- |
| 다변량 이상점수(주) | **Isolation Forest** (Liu·Ting·Zhou 2008) — 최근 주간 델타/기울기 벡터 |
| 보조 이상신호 | Local Outlier Factor (Breunig 2000), robust Mahalanobis/MCD (Rousseeuw 1999) |
| 변화점(급락 시점) | PELT/Binary Segmentation (Killick 2012, `ruptures`) |
| 신호별 추세·기울기 | Mann-Kendall + Theil-Sen (`pymannkendall`) |
| 궤적 군집화 | k-means → ‘급속 다변량 악화 / 만성 고위험 / 초기 변화 감지 / 안정·관찰’ |
| (Tier 2) DL | LSTM-AE (Malhotra 2016) — ⚠️ 합성 74×8 규모 과적합 → **미실행, 확장 경로로만 문서화** |

- ML은 오프라인에서 `households.json`에 `ml`(anomalyScore·clusterId·changePoints 등) 필드를 *구워서* 추가합니다. 런타임(브라우저)에선 모델이 돌지 않아 **client-only가 유지**되고, “점수는 투명 공식”(G1)·“결정은 사람”(G4)이 자연히 충족됩니다.
- **우선순위 = 위험점수 + 급속악화(이상점수) + 추세상승 + 반복통보** 의 **투명 융합**(블랙박스 랭커 아님 — 가중·규칙 전부 명시·조정가능).
- **분석 탭의 “결합 급락”**: 개별 신호는 모두 단일 임계 미만(임계값 방식은 놓침)이지만, 여러 신호가 동시에 최근 꺾여 다변량 이상탐지가 결합 이상으로 포착 → “사람이 그래프 하나씩 보면 되잖아”의 직접 반증.

## 모바일·PWA·인증 (작업 C)

- **반응형 + 하단 탭바** — 모바일에서 4탭을 엄지 도달 영역의 **하단 탭바**로, 카드형 reflow, 터치 타겟 ≥44px. **데스크톱 레이아웃은 그대로 유지**(상단 탭).
- **PWA** — `vite-plugin-pwa`로 매니페스트 + 서비스 워커 + 아이콘 생성 → 홈 화면에 **설치 가능한 앱**처럼 동작(오프라인 셸).
- **로그인/로그아웃** — `AuthProvider` + 라우트 가드 + 데모 공무원 계정(세션은 `localStorage`). 인증 백엔드는 **교체 가능하게 추상화**(`AuthBackend` 인터페이스 → 실제 OAuth/SSO로 전환 가능).
  - ⚠️ **정직성:** 데모용 인증입니다. **실제 보안 인증은 백엔드 연동(서버 세션·OAuth/SSO)이 필요**하며, 로그인 화면·본 문서에 명시합니다. 데모 계정: `kim`/`demo1234`, `lee`/`demo1234`.

## 한계와 책임

본 데모는 다음을 **명확히 구분**합니다.

**증명하는 것**

- 분석 파이프라인의 동작
- 위험 점수의 투명성·재현성
- 근거 기반 정보 구조가 의사결정 보조에 기여하는 방식(UX)

**증명하지 않는 것**

- 실제 발굴 정확도 및 예측 성능 (정답 레이블을 합성으로 설계했으므로 성능 평가는 순환적입니다)
- 정량적 발굴량 비교 ("몇 배" 등의 수치)

**ML·인증의 한계 (v3)**

- **ML은 보조 선별만** — 위험 *판정*이 아니라 급속·다변량 악화 *선별*과 우선순위 *그룹화*까지입니다. 헤드라인 점수는 투명 공식, 최종 결정은 담당자.
- **DL(Tier 2)은 미실행** — 합성 74가구×8주 규모에서 LSTM-AE 같은 DL은 과적합·시연용에 가까워, ‘실데이터 확장 경로’로만 문서화했습니다. 본체는 해석 가능한 Tier 1(Isolation Forest 등).
- **데모용 인증** — `localStorage` 기반 mock입니다. 실제 보안 인증은 백엔드(서버 세션·OAuth/SSO)가 필요하며, 인증 백엔드는 교체 가능하게 추상화했습니다.

**전제 및 윤리적 고려**

- 합성 데이터셋이며, 기존 시스템 기포착 여부(`haengbokFlagged`)는 시뮬레이션 플래그입니다.
- 실제 적용은 개인정보보호 법령, 지자체 위임, 실데이터 연동을 전제로 합니다.
- 모든 데이터는 가명처리·집계 분석을 전제로 하며, 재식별은 법적 통제 하에서만 수행되어야 합니다.
