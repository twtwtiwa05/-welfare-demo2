## 개요

기존 발굴 시스템은 다수의 위기정보를 융합해 위험가구를 선제적으로 발굴하지만, 산출된 명단이 **왜 위험한지**를 충분히 설명하지 못해 현장 담당자의 신뢰·활용으로 이어지지 않는 한계가 있습니다. 본 데모는 다음 세 지점에 집중합니다.

1. **재해석** — 단일 임계값(OR) 판정이 아니라 여러 신호의 가중 조합으로 위험을 평가하여, 개별 신호는 기준 미달이지만 동시에 위험 신호가 누적된 가구를 후보로 포착합니다.
2. **집단 특화** — 전국 단위 범용 모형 대신 대상 집단(독거노인 / 중장년 1인가구)별로 가중치를 분기하여, 집단 특수성을 반영합니다.
3. **활용도** — 점수와 신호를 근거로 한 자연어 설명·권고·우선순위·시계열 추적을 제공하여, 명단을 ‘행동’으로 전환합니다.

개선 지점은 동등한 과제가 아니라 **③ 활용도가 본체이고 ①·②는 이를 뒷받침**합니다. 화면은 4개 탭으로 이 위계를 반영하며, 활용도의 종착점으로 **방문 동선**(오늘 누구를, 어떤 순서·경로로)을 제시합니다.

## 핵심 설계 원칙

본 데모는 네 가지 설계 원칙을 코드 수준에서 강제합니다. 이는 시스템의 신뢰성·설명가능성·책임 소재를 보장하기 위한 제약입니다.

| 원칙 | 내용 | 구현 |
|---|---|---|
| **투명한 점수 산출** | 위험 점수는 가중합 공식이 계산합니다. 신경망/LLM이 점수를 만들지 않으며, 입력 변경 시 즉시 재계산되는 결정론적 함수입니다. | `src/lib/scoring.ts` |
| **역할이 한정된 AI** | 근거 생성기는 *이미 산출된* 점수와 신호를 입력받아 자연어 설명·권고만 생성합니다. 위험 여부나 점수를 판단하지 않습니다. | `src/lib/reason.ts` |
| **계산과 시뮬레이션의 명시적 구분** | 실측 계산 영역과 시뮬레이션(미리 정의된 값) 영역을 UI에 라벨링합니다. | `SimBadge`, `ProofBoundary` |
| **사람 중심 의사결정** | 최종 개입(전화·방문) 결정은 담당자가 합니다. AI는 발굴·근거·우선순위까지만 관여하며, ‘자동 처리’ 동작은 존재하지 않습니다. | `CaseDetail` (조치 ‘기록’만) |

## 주요 기능 (4개 탭)

상단에 **수원 모녀 사건 도입 훅**과 **증명 경계 패널**(증명/비증명/시뮬·합성)을 상설 배치합니다.

### 탭 1 — 분석 (받침 ①·②)
중복제거 요약 → **투명 점수 카드**(슬라이더 실시간 재계산 + 집단 모형 전환) → **임계값 vs 조합** 대조(거짓음성 관점). ③ 활용도를 신뢰하게 하는 최소 근거를 빠르게 제시합니다.

### 탭 2 — 활용도 Before / After (본체 ③) ★
동일한 명단을 **점수만 있는 표**(Before)와 **근거·출처 칩·추세·우선순위 번호·권고·반대근거가 결합된 카드**(After)로 대조합니다. 같은 데이터가 정보 구조에 따라 ‘행동 가능’해짐을 보여줍니다.

### 탭 3 — 방문계획 (③ 종착점) ★ 신규
**추세 반영 우선순위**(위험점수 + 최근 상승폭, 동점 시 반복 통보 우대)로 정렬하고, **근접 권역**으로 묶어 **방문 동선**(최근접 이웃 + 2-opt)을 지도에 번호 경로선으로 제시합니다. 순서·포함 여부는 담당자가 최종 결정합니다.

### 탭 4 — 케이스 관리 (운영 콘솔)
KPI 스탯, 검색·필터·CSV 내보내기, ‘오늘의 업무’ 워크리스트, 다중선택 일괄작업, 케이스 상세(기여도·근거·시계열·조치 기록·상태 워크플로우)를 제공합니다.

## 아키텍처

### 설계 개요
- **클라이언트 전용 SPA** — 서버·API 키가 필요 없어 임의의 정적 호스팅에 배포 가능합니다.
- **도메인 로직과 프레젠테이션 분리** — 점수·우선순위·근거 등 핵심 로직은 `src/lib`의 순수 함수로 구현되어 UI(`src/components`)와 독립적으로 테스트·재사용됩니다.
- **결정론적 파이프라인** — 동일 입력에 대해 항상 동일 출력을 반환합니다. 점수 모형과 근거 생성 모두 부수효과가 없습니다.

### 데이터 흐름
```
scripts/generate-data.mjs        (시드 고정 난수 → 합성 가구 생성, 좌표·반복통보 포함)
        │
        ▼
src/data/households.json         (정적 데이터셋, 74가구 · 좌표·repeatedFlags·8주 시계열)
        │
        ▼
src/lib/*  ── scoring · profiles · priority(추세 우선순위) · routing(동선) · reason · dedup · history
   (순수 도메인 로직: 정규화 → 가중합 → 구간 판정 → 우선순위 → 근접 묶음·경로)
        │
        ▼
src/components/*                 (React 프레젠테이션: 4개 탭 — 분석·활용도·방문계획·케이스 관리)
        │
        ▼
       UI
```

### 근거 생성 (LLM 역할)
데모에서는 LLM 호출을 **결정론적 템플릿**(`src/lib/reason.ts`)으로 시뮬레이션합니다. 점수·신호·케이스 유형으로부터 자연어 근거, 권고 2건, 반대 근거·불확실성, 근거 출처(신호 칩)를 구조화된 형태로 생성합니다. 실서비스 적용 시 이 레이어를 LLM API로 대체하되, 다음 제약을 유지하도록 설계되어 있습니다.

- 점수는 클라이언트의 투명 모형이 계산해 입력으로 전달하고, LLM은 점수를 재계산하지 않습니다(설명·서술만).
- API 키는 서버 측에서만 사용하고 브라우저에 노출하지 않습니다.
- 호출 실패 시 시뮬레이션 결과로 폴백합니다.

## 위험 점수 모형

각 신호를 0~1로 정규화한 뒤 집단별 가중치를 곱해 합산합니다(0~100 스케일).

```
정규화:
  power     = min(전력 사용 급감률 / 100, 1)
  medical   = min(진료 단절 일수 / 365, 1)
  mail      = min(우편 미수령 주 / 4, 1)
  telecom   = min(통신 연체 개월 / 3, 1)
  isolation = (복지관 6개월 이용 0회) ? 1 : 0

점수 = round( Σ (정규화_i × 가중치_i) )

위험 구간: 65점 이상 = 고위험 / 45점 이상 = 주의 / 그 외 = 관찰
```

**집단별 가중치 프로파일** (합 100) — 동일 가구라도 적용 모형에 따라 점수가 달라집니다.

| 신호 | 독거노인 (elderly) | 중장년 1인가구 (middleaged) |
|---|---|---|
| 전력 급감 (power) | 25 | 30 |
| 진료 단절 (medical) | 30 | 15 |
| 우편 미수령 (mail) | 25 | 15 |
| 통신 연체 (telecom) | 10 | 20 |
| 사회적 고립 (isolation) | 10 | 20 |

**발굴 방식 비교**
- 임계값 방식(기존): 단일 신호가 임계 초과 시 발굴 (`power≥80 || medical≥365 || mail≥4 || telecom≥3`)
- 조합 방식(본 데모): 가중합 점수 ≥ 50 이면 발굴

- Node.js 18 이상 (개발 환경 기준 22.x)

```bash
npm install        # 의존성 설치
```

산출물 `dist/`는 정적 파일이므로 Vercel·Netlify·Cloudflare Pages·GitHub Pages 등 임의의 정적 호스팅에 배포할 수 있습니다. 별도의 환경 변수나 서버 런타임이 필요하지 않습니다.

**Vercel 배포 예시**: 저장소를 import하면 Vite 프리셋이 자동 인식되며, 추가 설정 없이 빌드·배포됩니다. 이후 `main` 브랜치 푸시 시 자동 재배포됩니다.
